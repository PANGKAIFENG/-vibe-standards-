# Plan 命令

此命令调用 **planner** 代理在编写任何代码之前创建全面的实现计划。

## 此命令的功能

1. **重述需求** - 明确需要构建什么
2. **识别风险** - 揭示潜在问题和阻碍
3. **创建步骤计划** - 将实现分解为阶段
4. **等待确认** - 必须获得用户批准才能继续

## 何时使用

在以下情况使用 `/plan`：
- 开始新功能
- 进行重大架构变更
- 进行复杂重构
- 将影响多个文件/组件
- 需求不清晰或模糊

## 工作原理

planner 代理将：

1. **分析请求**并以清晰的术语重述需求
2. **分解为阶段**，包含具体可执行的步骤
3. **识别依赖**组件之间的关系
4. **评估风险**和潜在阻碍
5. **估算复杂度**（高/中/低）
6. **呈现计划**并等待您的明确确认

## 使用示例

```
用户: /plan 我需要在市场解析时添加实时通知

代理 (planner):
# 实现计划：实时市场解析通知

## 需求重述
- 在用户关注的市场解析时向用户发送通知
- 支持多个通知渠道（应用内、邮件、webhook）
- 确保通知可靠传递
- 包含市场结果和用户的仓位结果

## 实现阶段

### 阶段 1：数据库架构
- 添加 notifications 表，包含列：id, user_id, market_id, type, status, created_at
- 添加 user_notification_preferences 表用于渠道偏好
- 在 user_id 和 market_id 上创建索引以提升性能

### 阶段 2：通知服务
- 在 lib/notifications.ts 中创建通知服务
- 使用 BullMQ/Redis 实现通知队列
- 添加失败传递的重试逻辑
- 创建通知模板

### 阶段 3：集成点
- 接入市场解析逻辑（当状态变为 "resolved"）
- 查询所有在市场中有仓位的用户
- 为每个用户排队通知

### 阶段 4：前端组件
- 在 header 中创建 NotificationBell 组件
- 添加 NotificationList 模态框
- 通过 Supabase 订阅实现实时更新
- 添加通知偏好设置页面

## 依赖
- Redis（用于队列）
- 邮件服务（SendGrid/Resend）
- Supabase 实时订阅

## 风险
- 高：邮件可送达性（需要 SPF/DKIM）
- 中：每个市场有 1000+ 用户时的性能
- 中：如果市场频繁解析会产生通知垃圾
- 低：实时订阅开销

## 预估复杂度：中等
- 后端：4-6 小时
- 前端：3-4 小时
- 测试：2-3 小时
- 总计：9-13 小时

**等待确认**：继续执行此计划？(yes/no/modify)
```

## 重要说明

**关键**：planner 代理在您明确确认计划（如 "yes" 或 "proceed" 或类似肯定回复）之前**不会**编写任何代码。

如果您想要更改，请回复：
- "modify: [您的更改]"
- "different approach: [替代方案]"
- "skip phase 2 and do phase 3 first"

## 与其他命令集成

规划后：
- 使用 `/tdd` 以测试驱动开发方式实现
- 如果出现构建错误，使用 `/build-and-fix`
- 使用 `/code-review` 审查完成的实现
